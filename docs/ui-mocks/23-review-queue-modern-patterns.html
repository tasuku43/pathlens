Review queue UI exploration

Five modern patterns for the same Queued, In Review, Reviewed state model.

A Focus Stack B State Ledger C Rail + Preview D Command Dense E Quiet Table

All patterns keep the same product truth: unread is a small blue dot, queue rows do not repeat the state they are grouped under, the file tree and file header reuse the same state label, and Reviewed stays quiet until new change evidence moves a file back to Queued.

State first The user should know whether work is waiting, active, or already quiet before reading row detail.
One label system Queued, In Review, Reviewed appear in mixed surfaces only: file tree, tabs, and viewer header.
Low ceremony No separate Next Action card. The queue itself explains what can be done.

Pattern A

Focus Stack

Best default candidate. It keeps the current mental model but makes the hierarchy cleaner: counters, grouped sections, then rows.

  • Most faithful to the current implementation.
  • Readable at inspector width.
  • Good for mixed Git changes and open threads.

Pattern B

State Ledger

Turns the queue into a compact operational ledger. It is modern because every row is scannable and state is a column, not decoration.

  • Fastest for repeated review sessions.
  • Strong when there are many files.
  • Tradeoff: less warm than grouped sections.

Pattern C

Rail + Preview

A modern split inspector: left rail chooses the state, right side shows the current state's work. This makes the three states feel deliberate without becoming tabs.

  • Best when the inspector can be a little wider.
  • Great for teaching the state model.
  • Tradeoff: more layout machinery.

Pattern D

Command Dense

The most compact pattern. It treats the inspector like a worklist: strong keyboard posture, small counters, no large cards.

  • Best for power users and small screens.
  • Very close to CLI worklist mental models.
  • Tradeoff: less explanatory for first-time users.

Pattern E

Quiet Table

A polished table-like inspector. It is the least decorative and the most enterprise-operational: status, file, delta, evidence.

  • Best when accuracy and comparison matter most.
  • Easy to sort or virtualize later.
  • Tradeoff: less distinct from generic developer tools.
